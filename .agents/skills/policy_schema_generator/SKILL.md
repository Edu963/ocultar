# Policy Schema Generator (v2.1)

## Purpose

The Policy Schema Generator is the "Tier-0" transformer responsible for transcribing regulatory changes into the `regulatory_policy.json` schema. It ensures that every mapping in the Sombra Gateway layout is backed by a valid regulation and adheres to a strict `FAIL_CLOSED` security posture.

## Preconditions

- **Context**: Access to the `CISO & Compliance Officer` knowledge base.
- **Decoder**: `regulatory-intent-decoder` must have generated an `intent_manifest`.
- **Validation**: `jq` and `yq` must be available in the environment.
- **Safety**: Must have a clean state from `compliance-integrity-suite` before starting.

## Inputs / Outputs

### Inputs
- `intent_manifest` (Artifact): The structured output from `regulatory-intent-decoder`.
- `target_policy` (Path): Path to the existing `regulatory_policy.json` (default: `security/regulatory_policy.json`).
- `allow_downgrade` (Boolean): Explicit flag required to change `default_posture` from `FAIL_CLOSED` to `FAIL_OPEN`. (Default: `false`).

### Outputs
- `updated_policy` (File): Finalized `regulatory_policy.json`.
- `compliance_delta` (Artifact): A structured diff including:
    - Added/Modified categories.
    - Version bump details.
    - Regulation cross-references.
- `validation_report` (String): Success/Failure log from `jq` and `compliance-integrity-suite`.

## Instructions

### Phase 1 – Intent Ingestion
1.  **Load Manifest**: Read the `intent_manifest` generated by the `regulatory-intent-decoder`.
2.  **Refinery Verification**: Call `refinery-architecture-manager` (or read `refinery_manifest.json`) to confirm that the Refinery has an active regex/slm rule for every category in the manifest.
3.  **Halt on Mismatch**: If the Refinery cannot detect the category, fail with `UNSUPPORTED_DETECTION_CATEGORY`.

### Phase 2 – Configuration Logical Mapping
1.  **Read Current State**: Load `target_policy` into memory. 
2.  **Mapping Logic**:
    - **Update**: If category exists, update `regulation`, `article`, and `description` from the manifest.
    - **Insert**: If new, append key-value pair to the `mappings` object.
3.  **Posture Guard**:
    - If `default_posture` is being changed to `FAIL_OPEN` and `allow_downgrade` is `false`, **Reject Change**.

### Phase 3 – Semantic Versioning
1.  Compare the new mapping set with the old one.
2.  **Increment Logic**:
    - `Major`: If `default_posture` changes or > 30% of mappings are deleted.
    - `Minor`: If mappings are added or updated.
    - `Patch`: If only descriptions or metadata are refined.

### Phase 4 – Validation & Simulation
1.  **Structural Check**: Run `jq '.'` to ensure valid JSON syntax.
2.  **Simulation**: Call the `policy-impact-simulator` skill with the new JSON to generate a "Traffic Impact Report".
3.  **Halt on High Risk**: If the simulator returns a `HIGH` risk score, pause and request manual override.

### Phase 5 – Atomic Commit & Signing
1.  Write the new JSON to a temporary file (`/tmp/new_policy.json`).
2.  **Cryptographic Signing**: Call `compliance-certificate-signer` to sign the temporary file.
3.  Perform an atomic move to `target_policy`.
4.  **Post-Commit Sync**: Trigger `compliance-integrity-suite` to verify the state of the system.

---

## Failure Handling

- **`MANIFEST_MISSING`**: If the `intent_manifest` is not provided.
- **`DOWNGRADE_DENIED`**: If a posture shift is attempted without `allow_downgrade=true`.
- **`REFINERY_MISMATCH`**: If requirements exceed the Refinery's detection capabilities.
- **`SIMULATION_FAILURE`**: If the impact simulator predicts a system-wide outage.
